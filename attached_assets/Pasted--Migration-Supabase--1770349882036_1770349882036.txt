1) تنفيذ النقل — هدف نهائي (Zero Supabase)

بعد الموافقة:
هدفنا النهائي:

لا يوجد أي استخدام لـ supabase.from() أو supabase.rpc() أو Supabase DB نهائيًا.

نقل الـ Auth بالكامل إلى Express/Neon (جلسات/توكن) بحيث لا نحتاج supabase.auth في الـ frontend.

Edge Functions (supabase.functions.invoke) تظل مؤقتًا فقط لو كانت خاصة ZATCA/صور… لكن يتم عمل “Wrapper” لها داخل السيرفر لتقليل لمس Supabase من الواجهة.

2) تصميم Auth الجديد (B) — بدون كسر UI

نفّذ Auth على Express/Neon كالتالي:

DB Tables (إذا غير موجودة) + سيكيور:

auth_users (id uuid, email, username, password_hash, is_active, created_at…)

auth_sessions (id, user_id, token_hash, expires_at, created_at, revoked_at…)

فهارس مناسبة + unique على email و username

Endpoints:

POST /api/auth/register (email + username + password)

POST /api/auth/login (يدعم login بـ email أو username)

POST /api/auth/logout

GET /api/auth/me

استخدم HttpOnly cookie session أو Bearer token (اختر واحد ووضّح السبب).

لازم يكون فيه middleware requireAuth لحماية كل /api/** الحساسة.

Frontend:

استبدل AuthContext/AuthPage بالكامل ليستخدموا endpoints الجديدة.

احذف أي dependency على supabase.auth من الـ UI.

خلي تجربة الدخول والخروج stable + refresh page لا يضيع session.

مهم: لا تغيّر منطق الصلاحيات/roles—بس غيّر مصدر الـ auth.

3) Data-plane Migration (Neon/Express) — قاعدة صارمة

أي supabase.rpc() في الـ frontend تتحول إلى:

dataGateway.rpc(fnName, payload) → POST /api/rpc/:fnName

أي supabase.from().select() تتحول إلى:

dataGateway.queryTable(...) أو endpoints مخصصة في السيرفر حسب التعقيد.

ممنوع direct writes نهائيًا من الـ frontend:

كل writes لازم RPC atomic فقط.

نفّذ Migration بالـ Batches (بالترتيب):

Batch 1: shared libs + hooks

Batch 2: domain services

Batch 3: critical pages (Dashboard, Import, POS, JournalEntries, Purchasing hub)

Batch 4: reports

Batch 5/6: باقي components/pages

بعد كل Batch:

اعمل rg/grep evidence أن عدد supabase.from و supabase.rpc قلّ فعلاً.

شغّل smoke tests (load pages الأساسية) واطبع النتائج.
