نريد: POS Admin (username/password) يشتغل تلقائيًا ككاشير باسمه داخل POS.
يعني بعد Admin login + اختيار الفرع: cashier_user_id يتعين تلقائيًا = admin identity.
Admin له صلاحية مطلقة: بيع + مرتجع بدون Supervisor confirm + reports/settings.
مع الحفاظ على Audit: لا عمليات بيع/مرتجع بدون cashier_user_id.

المطلوب:

(1) DB/Identity
- pos_admin_accounts موجود.
أضف حقل:
display_name text not null
(أو استخدم username كاسم عرض لو مش هتضيف field)

(2) POS Admin Session + Branch Select (كما هو)
- POST /api/pos/admin/login (بدون requireBranchSession) => ينشئ __pos_admin session
- POST /api/pos/admin/select-branch (requires __pos_admin) => يثبت branch session للفرع المختار

(3) Admin يعمل كاشير باسمه (التغيير الأساسي)
A) Backend:
- عند نجاح /api/pos/admin/select-branch:
  - أنشئ/حدّث pos session للفرع المختار بحيث:
    pos_admin=true
    pos_admin_id=<admin.id>
    cashier_mode="admin"
    cashier_display_name = pos_admin.display_name (أو username)
    cashier_user_id (مهم): لا نملك ERP user uuid هنا، لذا:
      - خزّن admin كـ cashier في pos_sessions عبر حقول مخصصة:
        cashier_admin_id uuid (FK إلى pos_admin_accounts)
        cashier_user_id يبقى nullable لكن لا يُستخدم للأدمن
  - ثم عدّل كل عمليات البيع/المرتجع في POS لتقبل أحد الشكلين:
    - cashier_user_id (للكاشير العادي)
    - أو cashier_admin_id (للأدمن)
  - Enforcement:
    أي عملية بيع/مرتجع يجب أن تحتوي (cashier_user_id OR cashier_admin_id) وإلا reject.

B) Session Context:
- GET /api/pos/session/context يرجع:
  pos_admin=true/false
  pos_admin_id
  branch_id
  cashier_kind: "user" | "admin"
  cashier_name: string
  cashier_user_id (إن كان user)
  cashier_admin_id (إن كان admin)

(4) Frontend:
- في POS flow:
  - Admin بعد اختيار الفرع يدخل POS مباشرة بدون اختيار بائع وبدون PIN.
  - UI البائع يعرض:
    "البائع: <اسم الأدمن>"
    ولا يظهر "اختر بائع" للأدمن.
- الكاشير/المشرف: يبقى كما هو (اختيار بائع + PIN).

(5) Returns:
- إذا cashier_kind="admin" أو pos_admin=true:
  - نفّذ المرتجع بدون Supervisor confirm
  - سجّل performed_by_admin_id = pos_admin_id
