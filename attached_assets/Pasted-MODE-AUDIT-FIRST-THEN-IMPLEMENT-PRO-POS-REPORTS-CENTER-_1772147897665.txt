GOAL
Build a fully independent POS Reports Center at /pos/reports (POS Admin only), with a complete suite of POS reports. This Reports Center must NOT depend on the POS Dashboard content or its widget visibility settings. Dashboard and Reports are separate products.

NON-NEGOTIABLE (HARD RULES)
1) Do NOT touch ERP endpoints under /api/admin/* (leave them unchanged).
2) All new POS Reports endpoints MUST be under /api/pos/admin/reports/* and MUST use requirePosAdminToken.
3) Do NOT break existing POS flows (branch login gate, cashier gate, POS session).
4) NO GUESSING:
   - If a report needs data that does not exist, you must prove the gap with evidence and mark the report BLOCKED.
   - Only propose minimal additive changes AFTER proving gaps, and only via the repo’s official migration mechanism.
5) Keep response convention:
   - Success: { data: <payload>, error: null }
   - Failure: { data: null, error: { message: "..." } }
6) Reports Center MUST be independent:
   - /pos/reports must not read pos_dashboard_widgets or any dashboard settings.
   - Remove any dashboard→reports drill-down links if present.

PHASE B — IMPLEMENTATION (ONLY AFTER PHASE A IS ACCEPTED)

B1) Backend: Implement POS Reports endpoints (READ-ONLY) under /api/pos/admin/reports/*
All require requirePosAdminToken.

Minimum required endpoints (implement only those marked READY in Phase A):
R1 GET /api/pos/admin/reports/seller-performance?start=&end=&branch_id=&seller_id=
- Output per seller: sale_count, revenue, discounts, tax, net_total (define net_total with evidence)

R2 GET /api/pos/admin/reports/seller-net-profit?start=&end=&branch_id=&seller_id=
- revenue from sales_invoice_items totals
- cogs from unique_items.cost joined via sales_invoice_items.jewelry_item_id -> unique_items.id
- returns deduction derived via returns.original_sale_id + return_items + unique_items.cost
- output: revenue, cogs, gross_profit, return_amount, return_cost, net_profit, margin_percent

R3 GET /api/pos/admin/reports/branch-performance?start=&end=&branch_id=
R4 GET /api/pos/admin/reports/branch-net-profit?start=&end=&branch_id=
R5 GET /api/pos/admin/reports/sales-trend?start=&end=&branch_id=&seller_id=
- daily totals (date bucket) with count + revenue + net

R6 GET /api/pos/admin/reports/returns-summary?start=&end=&branch_id=&seller_id=
- grouped by derived seller + branch
- include top reasons if return_items.reason exists

R7 GET /api/pos/admin/reports/payment-mix?start=&end=&branch_id=&seller_id=
- grouped by payment_method: amount, count, percent

R8 GET /api/pos/admin/reports/reconciliation?start=&end=&branch_id=&limit=
- invoices.total_amount vs SUM(payments.amount) where payments.reference_type='invoice' AND reference_id=invoices.id
- return only rows with delta != 0

R9 GET /api/pos/admin/reports/inventory-valuation?branch_id=
- unique_items where status='in_stock'
- grouped by branch: item_count, total_cost_value, total_tag_value, potential_margin

R10 GET /api/pos/admin/reports/inventory-aging?branch_id=
- buckets by unique_items.created_at: 0-30,31-60,61-90,90+
- per-branch + overall totals

R11 GET /api/pos/admin/reports/sold-items-margin?start=&end=&branch_id=&seller_id=&limit=
- per unique item sold: serial_no, invoice_number, sale_date, branch, seller, sale_price, cost, margin, margin_percent

R12 GET /api/pos/admin/reports/inventory-ledger?start=&end=&branch_id=&limit=
- from unique_item_movements joined with unique_items and branches and reference fields

If Phase A proves more READY reports (VAT summary, outstanding aging, session summary), add them.

Validation rules:
- start/end required for date-based reports.
- branch_id/seller_id optional.
- limit optional, with safe default.

Performance:
- Add LIMIT on heavy lists.
- Avoid N+1 loops; do grouping in SQL.
- Use consistent ordering.
