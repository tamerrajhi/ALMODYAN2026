# Phase 2 — Server Endpoints (READ + ADMIN) using existing patterns in server/routes.ts
## 2A) READ: sellers list for a branch (used by POS)
GET /api/branches/:branchId/sellers
Return sellers allowed for that branch based on user_branches + profiles:
SQL must be server-side.

Result shape:
{
  data: { sellers: [{ profile_id, user_id, display_name, full_name, username }] },
  error: null
}

Rules:
- display_name = COALESCE(NULLIF(full_name,''), username)
- only active profiles (if is_active exists)
- order by display_name

## 2B) ADMIN: list branches + their sellers (used in /users tab)
GET /api/admin/branches-with-sellers
Return:
{
  data: { branches: [{ id, name, code, sellers: [...] }] },
  error: null
}

## 2C) ADMIN: list active profiles (for picker)
GET /api/admin/profiles/active
Return list of profiles (id, user_id, full_name, username, is_active)

## 2D) ADMIN: assign seller to branch
POST /api/admin/branch-sellers/assign
Body: { branch_id: uuid, profile_id: uuid, is_primary?: boolean }
Server resolves user_id from profiles.id, then upsert into user_branches:
INSERT ... ON CONFLICT(user_id, branch_id) DO UPDATE SET is_primary=EXCLUDED.is_primary;

Return success + updated seller row.

## 2E) ADMIN: remove seller from branch
POST /api/admin/branch-sellers/remove
Body: { branch_id: uuid, profile_id: uuid }
Resolve user_id then DELETE FROM user_branches.
