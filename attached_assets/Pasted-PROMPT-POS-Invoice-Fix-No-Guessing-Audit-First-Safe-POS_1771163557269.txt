PROMPT — POS Invoice Fix (No Guessing, Audit-First, Safe)

نفّذ إصلاح حسابات وعرض فاتورة البيع (POS Invoice / Receipt / PDF) بحيث تكون الحسابات صحيحة بدون أي تخمين وبدون أي تغيير في قاعدة البيانات أو الـRPC.

الهدف النهائي (المطلوب الصحيح)

في فاتورة البيع المطبوعة / المعروضة:

سعر كل قطعة في جدول الفاتورة
= السعر بعد الخصم وبدون ضريبة (NET).

الإجمالي (Subtotal)
= مجموع الأسعار بعد الخصم بدون ضريبة.

الضريبة
= محسوبة على الإجمالي بعد الخصم.

الإجمالي شامل الضريبة
= Subtotal + VAT.

أي:

الأسعار المعروضة للقطع = NET

الإجمالي = NET

الضريبة كسطر منفصل

ثم الإجمالي شامل الضريبة.

قواعد صارمة (ممنوع كسر السيستم)

❌ لا تعدل:

Database schema

migrations

RPC

backend logic

طريقة حفظ البيع

✔ التعديل Frontend فقط:

POS page

cart totals

receipt/print/pdf component

مرحلة 1 — Audit (إجباري قبل التعديل)

قبل أي تغيير:

1) اكتشف مصدر الحسابات الحالي

حدد:

أين تحسب totals؟

أين تحسب VAT؟

أين يحسب discount؟

أين يتم تجهيز بيانات الطباعة؟

2) أنشئ جدول Semantics في تعليق بالكود:

حدد لكل قيمة هل هي:

NET؟

أم GROSS (شامل الضريبة)؟

مثال:

line.price -> ?
line.total -> ?
cart.subtotal -> ?
vat -> ?
grandTotal -> ?


لا تفترض. اقرأ الحسابات الفعلية.

3) اطبع Debug Logs مؤقتاً

في وضع dev فقط:

netSubtotal
vatAmount
grossTotal
discount


ثم احذف أو علّق logs بعد التأكد.

مرحلة 2 — Fix الحسابات

في مكوّن الطباعة/الفاتورة:

جدول القطع

عمود السعر يجب أن يستخدم:

lineNetAfterDiscount


وليس السعر شامل الضريبة.

ملخص الفاتورة

يجب أن يكون:

Subtotal = sum(lineNetAfterDiscount)
VAT = subtotal * vat_rate
Grand Total = subtotal + VAT

ممنوع

عرض سعر شامل ضريبة كسعر القطعة.

حساب الضريبة مرتين.

خلط NET و GROSS في العرض.
